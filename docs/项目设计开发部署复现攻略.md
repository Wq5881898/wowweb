## 1. 文档目的

这份文档记录 WoW Register Lite 从需求提出、功能开发、AzerothCore 联调、邮件配置、Windows 服务化，到 Cloudflare Tunnel 公网调通的全过程。

完成本文步骤后，应能在另一台 Windows 服务器上复现以下系统：

- 玩家通过网页注册 AzerothCore 账号。
- 玩家登录网站并查看账号、角色和服务器状态。
- 玩家修改密码、绑定邮箱、通过邮件重置密码。
- GM 管理客户端下载链接和处理玩家问题。
- 网站、MySQL 和 Cloudflare Tunnel 随 Windows 自动启动。
- 用户通过固定 HTTPS 域名访问，不需要开放公网 80、443 或 8000 端口。

本文不会记录任何真实密码、SMTP 授权码或 Tunnel Token。迁移时必须重新填写自己的秘密配置。

## 2. 项目演进记录

### 2.1 原始目标

原始需求把项目定位为一个轻量 AzerothCore 注册入口：

1. Flask 提供注册页面。
2. 后端校验账号、密码和邮箱。
3. 后端通过 worldserver SOAP 执行固定的 `account create` 命令。
4. 不直接插入 `acore_auth.account`，避免自行处理账号创建和密码算法。
5. 第一版不做 CMS、商城、论坛和排行榜。

### 2.2 后续需求扩展

项目在第一版注册成功后逐步增加了：

- 登录页和用户 Dashboard。
- 注册时间、最后登录、角色数量和 GM 等级展示。
- 服务器状态和在线人数。
- 登录用户修改密码和绑定邮箱。
- GM 管理客户端下载链接，普通用户只读。
- 玩家问题提交、图片上传、处理状态和 GM 回复。
- 数学验证码和 IP 限流。
- 邮件密码找回和一次性重置链接。
- `.env` 秘密配置。
- Waitress 正式运行和 Windows 服务化。
- Cloudflare Tunnel 固定域名 HTTPS 公网访问。

因此，当前版本已经不是单纯注册页，而是一个轻量账号门户。

## 3. 当前系统架构

```text
玩家浏览器
    |
    | HTTPS: https://reg.151077.xyz
    v
Cloudflare 边缘网络
    |
    | Cloudflare Tunnel（服务器主动向外建立连接）
    v
cloudflared Windows 服务
    |
    | HTTP: http://127.0.0.1:8000
    v
Waitress / WowWeb Windows 服务
    |
    +-- Flask 页面、会话、业务逻辑
    +-- MySQL 3306 -> acore_auth / acore_characters
    +-- SOAP 7878 -> AzerothCore worldserver
    +-- SQLite -> data/wowweb.db
    +-- SMTP -> 密码重置邮件
```

### 3.1 各组件职责

| 组件 | 职责 |
| --- | --- |
| Flask | 路由、表单校验、登录会话和业务逻辑 |
| Waitress | 正式环境 WSGI 服务，相当于 Python Web 应用的生产容器 |
| WinSW | 把 Waitress 命令包装成 Windows 服务 |
| MySQL | AzerothCore 账号、权限、角色和在线状态数据 |
| worldserver SOAP | 执行受控 AzerothCore 命令，主要用于注册和服务器状态查询 |
| SQLite | 保存下载链接、玩家问题、GM 回复和密码重置 Token |
| cloudflared | 把 Cloudflare 的公网 HTTPS 请求转发到本机 Waitress |
| Cloudflare DNS/Tunnel | 固定域名、TLS 证书、公网入口和代理 |

### 3.2 数据访问边界

账号创建始终通过 SOAP：

```text
account create <username> <password> <email>
```

增加登录、修改密码和密码找回后，应用开始受控访问 MySQL：

- 从 `acore_auth.account` 读取账号资料、salt 和 verifier。
- 从 `acore_auth.account_access` 读取 GM 等级。
- 按 AzerothCore SRP6 规则校验或更新密码。
- 更新账号邮箱。
- 从 `acore_characters.characters` 读取角色数和在线角色。

这意味着迁移时必须确认目标 AzerothCore 版本的账号表结构和 SRP6 实现与当前代码兼容。账号创建仍不直接写数据库。

## 4. 当前功能清单

### 4.1 页面与接口

| 地址 | 方法 | 权限 | 功能 |
| --- | --- | --- | --- |
| `/` | GET | 公开 | 登录、注册入口和数学验证码 |
| `/register` | POST | 公开 | 校验并通过 SOAP 创建账号 |
| `/login` | POST | 公开 | 校验 AzerothCore 账号并建立会话 |
| `/logout` | POST | 登录 | 清除会话 |
| `/dashboard` | GET | 登录 | 账号、角色数量、服务器状态和 realmlist |
| `/account/security` | GET/POST | 登录 | 修改密码 |
| `/account/email` | POST | 登录 | 验证当前密码后绑定或更新邮箱 |
| `/forgot-password` | GET/POST | 公开 | 校验账号和邮箱并发送重置邮件 |
| `/reset-password/<token>` | GET/POST | 公开 | 使用一次性 Token 重置密码 |
| `/downloads` | GET | 登录 | 查看客户端下载链接 |
| `/downloads` | POST | GM | 新增、编辑或删除下载链接 |
| `/online` | GET | 登录 | 查询当前在线角色 |
| `/issues` | GET/POST | 登录 | 提交和查看问题，可上传图片 |
| `/issues/<id>/moderate` | POST | GM | 更新问题状态和 GM 回复 |

### 4.2 本地 SQLite 表

应用首次启动时自动创建 `data/wowweb.db`：

- `download_links`：标题、URL、说明和更新时间。
- `issues`：用户、内容、图片路径、状态和 GM 回复。
- `password_reset_tokens`：Token 哈希、过期时间、使用时间和请求 IP。

问题状态包括：`open`、`in_progress`、`resolved`、`closed`。

### 4.3 安全措施

- 账号名仅允许 3 到 16 位字母、数字和下划线。
- 密码限制为 6 到 32 位且不能包含空格。
- 注册邮箱必填并校验基本格式。
- SOAP XML 对命令内容进行转义。
- 应用只拼接固定格式的受控命令，不接受用户自定义 GM 命令。
- 注册和密码重置按 IP 做内存限流。
- 注册使用服务端数学验证码。
- 上传文件使用随机 UUID 文件名并限制扩展名和大小。
- 重置 Token 只保存 SHA-256 摘要，具有有效期且只能使用一次。
- 找回密码始终返回通用提示，避免泄露账号是否存在。
- 日志滚动保存，不记录密码。

## 7. 配置 AzerothCore SOAP

### 7.1 启用 SOAP

编辑 `worldserver.conf`：

```ini
SOAP.Enabled = 1
SOAP.IP = "127.0.0.1"
SOAP.Port = 7878
```

重启 worldserver 后检查：

```powershell
Get-NetTCPConnection -LocalPort 7878 -State Listen
```

必须只看到本机地址。不要把 7878 映射到公网，也不要添加公网防火墙规则。

### 7.2 创建专用 SOAP 账号

在 worldserver 控制台创建独立账号并授予足够的 GM 权限。命令格式会因 AzerothCore 版本略有区别，常见形式为：

```text
account create websoap <强随机密码>
account set gmlevel websoap 3 -1
```

该账号只供网站调用 SOAP，不用于正常游戏。配置后应先测试 `server info`，再测试临时账号的 `account create`。

### 7.3 SOAP 排障

| 现象 | 原因与处理 |
| --- | --- |
| 连接被拒绝 | worldserver 未启动、SOAP 未启用或端口不一致 |
| HTTP 401/403 | SOAP 用户名或密码错误 |
| permission/security 错误 | SOAP 账号 GM 权限不足 |
| 请求超时 | worldserver 阻塞、SOAP 无响应或本机安全软件拦截 |
| 网页可开但不能注册 | Waitress 正常不代表 SOAP 正常，单独检查 7878 |

## 8. 配置数据库权限

`.env` 中的 MySQL 用户至少需要：

- `acore_auth.account`：`SELECT`、`UPDATE`。
- `acore_auth.account_access`：`SELECT`。
- `acore_characters.characters`：`SELECT`。

不要为了方便让网站使用 MySQL root。若数据库不在同一台机器，应限制来源 IP并启用安全连接；同机部署优先使用 `127.0.0.1`。

## 10. 邮件密码找回配置记录

### 10.1 方案选择过程

最初尝试 Outlook SMTP，但个人账号、应用密码、SMTP AUTH 和微软租户策略之间存在限制；Azure 应用注册页面也提示需要 M365 开发者计划或目录权限。对于这个小型站点，该路线配置成本偏高。

最终改用 163 邮箱 SMTP：

- SMTP 服务器：`smtp.163.com`
- SSL 端口：`465`
- `SMTP_USE_SSL=true`
- `SMTP_USE_TLS=false`
- 登录密码使用 163 邮箱设置中生成的“客户端授权密码”，不是网页登录密码。

POP3 用于收邮件，不能代替 SMTP 发信。即使邮箱后台同时开启 POP3/SMTP，网站实际使用的仍是 SMTP。

### 10.2 功能约束

- 新注册用户必须填写邮箱。
- 旧账号必须登录后在“账号安全”绑定邮箱。
- 找回时必须同时输入账号和已绑定邮箱。
- 重置链接由 `PUBLIC_BASE_URL` 生成，正式环境必须填写公网 HTTPS 地址。
- Token 默认 30 分钟过期，使用后立即失效。

### 10.3 邮件测试

1. 注册测试账号并确认数据库中邮箱正确。
2. 打开 `/forgot-password`，输入账号和邮箱。
3. 检查收件箱和垃圾邮件。
4. 打开邮件链接并修改密码。
5. 再次打开旧链接，应提示无效或过期。
6. 使用新密码登录网站和游戏。

## 14. 公网网络方案的选择过程

### 14.1 最初的 Caddy 方案

最初计划使用：

```text
公网 80/443 -> 路由器端口映射 -> Caddy -> 127.0.0.1:8000
```

Caddy 可以反向代理并通过 Let's Encrypt 自动申请证书。但公网运营商可能封锁入站 80/443，导致：

- 用户无法连接源站 443。
- HTTP-01/TLS-ALPN-01 域名验证失败。
- 即使改用 `https://域名:8443`，用户仍需显式输入端口，且证书签发问题不会自动消失。

之后研究了 DNS-01。DNS-01 不要求开放 80/443，但 Caddy 需要 DNSPod 插件和 API Token，才能在续期时自动创建 `_acme-challenge` TXT 记录。该插件只解决证书所有权验证，不能解决用户无法连接源站端口的问题。

因此最终推翻 Caddy 方案。

### 14.2 最终的 Cloudflare Tunnel 方案

Cloudflare Tunnel 的关键优势：

- cloudflared 从服务器主动向 Cloudflare 建立出站连接。
- 家庭网络无需开放或映射 80、443、8000、8443。
- 用户仍通过标准 `https://域名` 访问。
- Cloudflare 负责公网 TLS 证书。
- 固定自定义域名不会像临时 Quick Tunnel 域名那样变化。

当前 Caddy 服务 `WowWebCaddy` 已停止并禁用，80/443/8443 的旧防火墙放行规则已删除。不要同时启用两套入口，避免排障混乱。

## 15. Cloudflare Tunnel 配置

### 15.1 域名必须由 Cloudflare 托管 DNS

要给 Tunnel 使用 `reg.151077.xyz`，根域 `151077.xyz` 必须添加到 Cloudflare，并在域名注册商处把权威名称服务器改成 Cloudflare 分配的两条 NS。

修改 NS 前，应先把原 DNSPod 中仍在使用的 A、CNAME、MX、TXT 等记录完整复制到 Cloudflare。改变 NS 后，DNSPod 中的记录不会再对公网生效，但可以保留作为历史配置。

同一个根域下可以有多个子域指向不同服务器。例如原有 `www` 或根域 A 记录可以继续服务其他系统，新增的 `reg` 单独指向 Tunnel，互不冲突。

### 15.2 创建 Tunnel 和公开主机名

Cloudflare 控制台路径通常为：

```text
Zero Trust -> Networks -> Tunnels
```

创建 Tunnel 后，在 Published application / Public hostname 中填写：

| 字段 | 值 |
| --- | --- |
| Subdomain | `reg` |
| Domain | `151077.xyz` |
| Path | 留空 |
| Service type | `HTTP` |
| Service URL | `http://127.0.0.1:8000` |

保存后，Cloudflare 会创建一条代理 CNAME/Tunnel DNS 记录，目标类似：

```text
<tunnel-uuid>.cfargotunnel.com
```

不要手工把该记录改成源站公网 IP。

### 15.3 安装 cloudflared 服务

在 Tunnel 安装页面选择 Windows，复制 Cloudflare 给出的安装命令，在管理员 PowerShell 执行。远程托管 Tunnel 的常见形式是：

```powershell
cloudflared.exe service install <TUNNEL_TOKEN>
```

Token 等同于连接 Tunnel 的凭据，不得提交到 Git、截图公开或写进本攻略。安装后服务命令会类似：

```text
cloudflared.exe tunnel run --token <REDACTED>
```

### 15.4 验证服务和公网

```powershell
Get-Service Cloudflared
Get-CimInstance Win32_Service -Filter "Name='Cloudflared'" |
    Select-Object Name,State,StartMode,ProcessId

Resolve-DnsName -Type NS 151077.xyz
Resolve-DnsName reg.151077.xyz
Clear-DnsClientCache
curl.exe -I https://reg.151077.xyz/
```

预期：

- `Cloudflared` 为 `Running`、`Automatic`。
- NS 是 Cloudflare 分配的名称服务器。
- `reg` 解析到 Cloudflare 代理 IP，而不是家庭公网 IP。
- 公网请求返回 HTTP 200 或正常跳转。

2026-06-24 实测 `https://reg.151077.xyz/` 返回 HTTP 200。

## 16. NXDOMAIN 和 Tunnel 排障攻略

### 16.1 已遇到的问题

Cloudflare 控制台中已经创建 `register.151077.xyz`，之后改成 `reg.151077.xyz`，Tunnel 目标也显示正确，但浏览器仍报：

```text
DNS_PROBE_FINISHED_NXDOMAIN
```

### 16.2 根因

当时域名仍由 DNSPod 的 NS 负责权威解析。记录虽然存在于 Cloudflare 控制台，但公网 DNS 查询根本不会询问 Cloudflare，所以返回 NXDOMAIN。

这与以下内容无关：

- Waitress 是否启动。
- Service URL 是否为 `127.0.0.1:8000`。
- CNAME 的 Tunnel UUID 是否正确。
- cloudflared 服务是否在线。

DNS 尚未解析成功时，请求还没有到 Cloudflare，更没有到服务器。

### 16.3 标准排障顺序

严格按层检查：

1. `Resolve-DnsName -Type NS 根域`：确认权威 DNS 是 Cloudflare。
2. `Resolve-DnsName 子域`：确认子域存在并解析到 Cloudflare。
3. Cloudflare 控制台：Tunnel 状态应为 Healthy/Connected。
4. `Get-Service Cloudflared`：本机服务应运行。
5. `curl http://127.0.0.1:8000/`：确认源站正常。
6. `curl https://公网域名/`：最后检查完整链路。

不同错误对应不同层：

| 错误 | 优先检查 |
| --- | --- |
| `NXDOMAIN` | 权威 NS、DNS 记录和缓存 |
| Cloudflare 1033 | Tunnel 未连接或 Connector 离线 |
| Cloudflare 502 | Tunnel 在线，但 `127.0.0.1:8000` 不可访问 |
| 本机 8000 拒绝连接 | WowWeb 服务、Waitress 日志和 `.env` |
| 首页正常但注册失败 | worldserver、SOAP 7878 和 SOAP 凭据 |

## 17. Windows 防火墙和端口结论

当前 Tunnel 架构不需要公网入站端口映射：

| 端口 | 用途 | 建议 |
| --- | --- | --- |
| 8000 | Waitress | 只监听 `127.0.0.1`，不开放公网 |
| 7878 | worldserver SOAP | 只监听 `127.0.0.1`，不开放公网 |
| 3306 | MySQL | 同机时仅本机访问，不开放公网 |
| 80/443 | 传统反代入口 | Tunnel 方案不需要在源站开放 |
| 8443 | 曾考虑的 Caddy HTTPS | 当前不使用，不开放 |

cloudflared 需要正常的出站 HTTPS/QUIC 网络。如果网络阻止 QUIC，它通常可以回退到 HTTP/2；排障时检查安全软件和出站策略。
